# テスト実行手順書 (Phase 12: Staging最終統合)

## 概要
本ドキュメントは、GitHub Issue #30 に基づく最終統合機能（動的メール認証、プロフィール編集、詳細プラン管理、実データ表示）を検証するための手順を定義します。

---

## 1. 動的メール認証の検証 (Sign Up)

### 1.1 認証コードの取得
1. ブラウザで `/signup` を開き、ステップ2まで進みます。
2. 有効なメールアドレスを入力して「送信」をクリックします。
3. **バックエンドのログ** を確認し、生成されたコードを特定します。
```bash
docker compose logs backend | grep "Verification Code"
```
**期待結果**: 「[Verification Code] for [email]: 123456」のようなログが出力されていること。

### 1.2 コードの検証
1. ログで確認した6桁のコードを画面に入力します。
**期待結果**: 認証が成功し、ユーザー情報入力（ステップ3）へ進めること。

---

## 2. アカウント・プロフィール編集の検証

### 2.1 プロフィールの更新
1. ログイン後、`http://localhost:3001/account` を開きます。
2. ニックネームまたはメールアドレスを書き換え、「保存」をクリックします。
**期待結果**:
- 「プロフィールを更新しました」というメッセージが表示されること。
- 画面をリロードしても、変更後の情報が保持されていること。

### 2.2 メールアドレス重複チェック
1. 既に存在する他のユーザーのメールアドレスに変更して保存を試みます。
**期待結果**: 「このメールアドレスは既に使用されています」等のエラーが表示されること。

---

## 3. 詳細プラン (Tier) の検証

### 3.1 プラン種別の同期
1. `stripe trigger customer.subscription.created` を実行（price_id に対応するものを指定）。
2. DB または `/api/user/profile` を確認します。
**期待結果**: `plan_tier` が `light`, `basic`, `pro` のいずれかに更新されていること。

---

## 4. 外部コンテンツの表示検証

### 4.1 Instagram 投稿の表示
1. Instagram 連携済みの状態で `/post` 画面を開きます。
2. `MOCK_MODE=false` であることを確認。
**期待結果**: プレースホルダーではなく、API から取得された実際の投稿写真が表示されること。

---

## 5. トラブルシューティング

### 5.1 認証コードがログに出ない
- **原因**: 環境変数 `MOCK_MODE=true` になっている、または送信ロジックのエラー。
- **対処**: `.env` の `MOCK_MODE` を確認。バックエンドのログで `POST /api/auth/send-code` が 200 を返しているか確認してください。

### 5.2 プロフィールが保存されない
- **原因**: ログインセッションの期限切れ、または DB マイグレーション未実施。
- **対処**: `docker compose exec backend ./server migrate up` を実行して、`users.nickname` カラムが存在するか確認してください。
