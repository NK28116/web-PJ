## レビュー日: 2026-03-12

### 1. 総評
Phase 5-1「OAuth基盤実装」は、実運用に耐えうる堅牢なアーキテクチャで完遂されています。特に、暗号化されたトークンの永続化と、リクエスト直前の自動リフレッシュ機構が統合されており、開発者が「トークンの期限」を意識せずにビジネスロジックに集中できる環境が整っています。これは次フェーズのレポート機能実装において、極めて強力な武器となります。

---

### 2. 項目別詳細

#### A. OAuth 認可フローとセキュリティ (✅ 優秀)
- **CSRF 対策**: `state` パラメータの生成と Cookie による検証が、Google/Instagram 両方のハンドラで徹底されています。
- **暗号化プロトコル**: `AES-256-GCM` を用いたアクセストークン・リフレッシュトークンの暗号化により、DB 流出時のリスクを最小限に抑えています。
- **Scope の網羅性**: 
    - Google: `business.manage` (Business Profile API 用)
    - Instagram: `instagram_manage_insights`, `pages_read_engagement` (インサイト取得用)
    これらが既に確保されており、追加の認可フローを回すことなくレポート機能の実装に入れます。

#### B. トークン・オーケストレーション (Service層) (✅ 合格)
- **`token_refresh.go`**: 
    - Google (OAuth2 Standard) と Instagram (Long-lived Token API) の異なる更新プロトコルが、一つのインターフェース (`RefreshTokenIfNeeded`) の背後に美しく隠蔽されています。
    - 有効期限の 5 分前に更新をトリガーする「プロアクティブ・リフレッシュ」により、API 呼び出し中の期限切れエラーを未然に防いでいます。

#### C. DB 永続化とデータ整合性 (✅ 合格)
- **`ExternalAccount` モデル**: プロバイダーごとの ID (`provider_user_id`) と、認可されたスコープ情報が保存されており、どのユーザーがどの権限を持っているかを正確に把握可能です。
- **Upsert ロジック**: 同一ユーザー・同一プロバイダーでの「再連携」時に、既存レコードをクリーンに更新する実装となっており、データの重複を防いでいます。

### 3. 次フェーズ (5-2) に向けた技術的提言

1.  **API クライアントの抽象化**:
    現在、`token_refresh.go` が直接 `http.Get` を呼んでいる箇所があります。Phase 5-2 では、リフレッシュロジックを内包した「外部 API 実行用ラッパー（Service クライアント）」を実装し、各ハンドラからはそのクライアントを呼ぶだけの形にすることを推奨します。
2.  **レートリミットの考慮**:
    レポート取得は大量の API コールを伴う可能性があります。各プロバイダーのクォータ制限を考慮し、必要に応じて結果を短期間キャッシュするロジックの導入を検討してください。
3.  **エラーのユーザー通知**:
    リフレッシュトークンが無効化された場合（ユーザーが外部サービス側で連携を解除した等）、システム側で「再連携が必要」であることをユーザーに通知するための、DB へのエラーフラグ更新処理を追加するとより親切です。
