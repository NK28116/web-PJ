# フェーズ別レビュー (Phase 5: OAuth基盤 & トークン管理)

## レビュー日: 2026-03-10
## レビュアー: 河崎 紗夜

### 1. 総評
OAuth 基盤の「保存・復号・自動更新」という一連のライフサイクル管理が実装されました。暗号化されたトークンの DB 永続化から、期限切れに伴う自動リフレッシュまで、バックエンドの主要なロジックが統合されています。設計は疎結合（Handler → Service → Repository）に保たれており、保守性の高い構成です。

---

### 2. 項目別詳細

#### A. データの永続化 (Repository層) (✅ 合格)
- **`external_account.go`**: `[]byte` 型の暗号化トークンを正しく処理。`Upsert` ロジック（既存アカウントがあれば更新、なければ作成）が実装されており、再連携時の挙動も安全です。
- **インターフェース化**: `interfaces.go` に定義が追加されており、将来的なテスト（モック化）が容易です。

#### B. トークン・リフレッシュ (Service層) (✅ 優秀)
- **`token_refresh.go`**: 
    - 期限切れの 5 分前（バッファ）に更新を試みるなど、実運用を考慮したロジックが組み込まれています。
    - 暗号化キーを用いた「復号 → OAuthリフレッシュ → 再暗号化 → 保存」のフローが正確に実装されています。

#### C. セキュリティ (✅ 優秀)
- 暗号化キー (`ENCRYPTION_KEY`) の扱いや、メモリ上での一時的な平文トークンの処理において、不必要な露出を避ける設計が継続されています。

---

### 3. 指摘・改善推奨事項 (Next Steps)

1.  **Instagram 特有のリフレッシュロジック**:
    Instagram の長期トークン（60日）のリフレッシュは Google の `refresh_token` とは API エンドポイントが異なります。現在共通化されていますが、Provider ごとに微調整が必要になる可能性があるため、テストでの検証を推奨します。
2.  **API 呼び出しの共通化**:
    各ハンドラでリフレッシュロジックを呼ぶのではなく、外部 API クライアントを初期化する際に「自動リフレッシュ機能付きの `http.Client`」を生成するユーティリティを作成すると、コードの重複が減り、より堅牢になります。
3.  **エラーハンドリング**:
    リフレッシュトークン自体が無効（ユーザーが外部サービス側でアプリ連携を解除した場合など）の際に、DB から古いトークンを削除する、またはユーザーに再認証を促す通知フラグを立てる処理の追加を検討してください。
